# LoggerLib (библиотека логирования)

---

## О проекте
Тестовое задание: библиотека логирования на C++17 с поддержкой записи в файл и в сокет
(взаимозаменяемые backend'ы через паттерн Strategy), консольное многопоточное приложение
для проверки библиотеки, и консольный сервер сбора статистики, принимающий логи по сокету.

Проект состоит из трёх независимых частей:

- **`lib/`** — библиотека логирования (статическая/динамическая сборка на выбор)
- **`app/`** — консольное приложение: читает сообщения из stdin, пишет через `Logger`
  в отдельном потоке, потокобезопасно
- **`stats_server/`** — консольный сервер: принимает логи по TCP от `Logger` с сокет-backend'ом,
  выводит их в консоль и считает статистику (по количеству, по уровням, по длине сообщений)

## Сборка

Зависимости:
- компилятор с поддержкой C++17 (проверено на gcc)
- CMake ≥ 3.10
- POSIX-совместимая ОС (проверено на Ubuntu/Debian) — используются POSIX sockets и `pthread`

Библиотека может быть собрана как статическая (по умолчанию) или как динамическая — переключается флагом `BUILD_SHARED_LIBS`:

```bash
# статическая сборка (по умолчанию)
cmake -B build -S .
cmake --build build

# динамическая сборка
cmake -B build -S . -DBUILD_SHARED_LIBS=ON
cmake --build build
```

Библиотека и приложение собираются как раздельные CMake-таргеты в одном проходе; после сборки исполняемый файл приложения находится в `build/app/`, сервера по сбору статистики `build/stats_server/`, библиотека — в `build/lib/`.

По умолчанию используется `Release`-сборка (оптимизации включены, `assert` отключены). Для отладочной сборки:

```bash
cmake -B build -S . -DCMAKE_BUILD_TYPE=Debug
cmake --build build
```

## Использование библиотеки

#### Инициализация с записью в файл:

```cpp
#include <logger/logger.hpp>

Logger logger("app.log", MessageLevel::INFO);

logger.log("Server started");                      // с уровнем по умолчанию (INFO)
logger.log("Connection refused", MessageLevel::ERROR);

logger.change_default_level(MessageLevel::WARNING); // сообщения ниже WARNING больше не пишутся
```

#### Инициализация с записью в сокет 
(тот же интерфейс `log()`, другой backend передаётся при создании)

```cpp
Logger logger("127.0.0.1", 9000, MessageLevel::DEBUG);
logger.log("Connected to stats collector");
```

Если файл/сокет не удалось открыть — `Logger` не бросает исключение (по требованиям ТЗ), 
а выводит причину в `stderr` при создании и на каждой последующей попытке записи; 
сама программа продолжает работать.

#### Прямое использование стратегий

Конкретные реализации `ILogStrategy` (`FileLogStrategy`, `SocketLogStrategy`) 
— часть публичного API и находятся в `include/logger/strategies/`. 
Это позволяет собрать и проверить стратегию до создания `Logger`, 
если нужен более тонкий контроль над обработкой ошибок инициализации:

```cpp
#include <logger/strategies/file_log_strategy.hpp>

std::string err;
auto strategy = FileLogStrategy::create("app.log", err);
if (!strategy) {
    // своя обработка ошибки — например, fallback на другой файл или завершение программы
}

Logger logger(std::move(strategy), MessageLevel::INFO);
```

## Использование консольного приложения

Запуск:

```bash
./app <log_file_name> <default_level> 
```

- `log_file_name` — путь к файлу журнала
- `default_level` — уровень важности по умолчанию: `DEBUG`, `INFO`, `WARNING` или `ERROR`

Пример:

```bash
./app app.log INFO
```

После запуска приложение построчно читает сообщения из stdin. Формат ввода:

```bash
<LEVEL> <message>     # явное указание уровня, например: ERROR Connection refused
<message>             # без уровня — используется default_level, например: Server started
```

Уровень — необязательный **первый токен** строки. Если первое слово строки точно совпадает с одним из `DEBUG`/`INFO`/`WARNING`/`ERROR`, оно интерпретируется как уровень, а остаток строки — как текст сообщения; в противном случае вся строка целиком считается сообщением с уровнем по умолчанию.

Каждое сообщение передаётся в отдельный поток для потокобезопасной записи в журнал; ввод не блокируется на время записи.

Завершение работы — `Ctrl+D` (EOF) / `Ctrl+C` / Введение пустой строки.

## Использование сервера статистики

Запуск:

```bash
./stats_server <host> <port> <T> <N>
```

- `host` — адрес для прослушивания (например, `127.0.0.1` или `0.0.0.0`)
- `port` — TCP-порт (`0..65535`)
- `T` — интервал в секундах между проверками статистики по таймеру; выводится, только если
  статистика изменилась с прошлого вывода
- `N` — статистика выводится дополнительно после каждого N-го принятого сообщения (суммарно
  по всем подключённым клиентам)

Пример:

```bash
./stats_server 127.0.0.1 9000 30 10
```

Сервер принимает несколько клиентов одновременно (по потоку на подключение), каждое принятое
сообщение выводится в консоль сразу, статистика — общая по всем источникам:

- общее число сообщений и число по каждому уровню важности
- число сообщений за последний час (скользящее окно)
- минимальная/максимальная/средняя длина текста сообщения (за всё время)

Строки, которые не удалось распарсить в ожидаемом формате `[LEVEL] [timestamp] [text]`,
пропускаются с предупреждением в `stderr`; соединение при этом не прерывается.

Завершение работы — `Ctrl+C` (`SIGINT`) или `SIGTERM`: сервер прекращает принимать новые
подключения, штатно завершает все активные клиентские соединения и потоки, затем выходит.

## Архитектурные решения

**Strategy для backend'ов логирования.** `Logger` не знает, пишет ли он в файл или в сокет —
оба варианта реализуют общий интерфейс `ILogStrategy`. Это отвечает требованию ТЗ о том, что
интерфейс логирования через сокет не должен отличаться от интерфейса логирования в файл,
и позволяет добавить новый backend в будущем без изменения `Logger`.

**Без исключений.** Все операции, которые могут завершиться неудачей (открытие файла,
подключение сокета, парсинг ввода), возвращают `std::optional`/`nullptr` вместо исключений.
Стратегии и `Logger` создаются через статические фабричные методы (`create()`), возвращающие
`nullptr` при неудаче, а не через конструктор, который может оставить объект в частично
рабочем состоянии — так поддерживается инвариант "если конструктор завершился, объект
полностью пригоден к использованию".

**Один worker-поток в консольном приложении.** Часть 2 использует один выделенный поток для
записи в журнал (не пул) — сообщения должны сохранять порядок поступления при записи в общий
файл, а сама запись — I/O-bound операция, не выигрывающая от распараллеливания на несколько
потоков.

**Поток на клиента в сервере статистики.** Сервер обслуживает нескольких одновременных
источников логов, аналогично модели "несколько источников — общая агрегация" (как в системах
мониторинга). Статистика — общий разделяемый ресурс с собственным мьютексом; вывод в консоль
защищён отдельным мьютексом, чтобы вывод сообщения не блокировал параллельное обновление
статистики другими клиентами.

**Graceful shutdown.** И консольное приложение, и сервер статистики используют RAII-обёртки
над потоками (деструктор сам сигналит остановку и дожидается `join()`), поэтому корректное
завершение работы не требует ручного управления в вызывающем коде. Сервер дополнительно
использует `shutdown(fd, SHUT_RDWR)` на все активные клиентские сокеты при получении
`SIGINT`/`SIGTERM`, чтобы разбудить клиентские потоки, блокирующиеся на `recv()`.

## Тестирование

Юнит-тесты используют GoogleTest (подключается автоматически через CMake `FetchContent`,
требует доступ в сеть на этапе конфигурации). Отключить сборку тестов можно флагом
`-DBUILD_TESTS=OFF`.

Запуск:

```bash
cmake --build build
ctest --test-dir build --output-on-failure
```

Покрыты: парсинг уровня и timestamp'ов, `LogEntry::parse`, `Statistics` (включая
конкурентное обновление под нагрузкой из нескольких потоков), `Logger` через
mock-реализацию `ILogStrategy`, разбор аргументов командной строки обоих приложений,
форматирование вывода статистики.

### Демонстрационный клиент для сервера статистики

`tests/manual/socket_demo_client` — не юнит-тест, а вспомогательный инструмент для ручной
проверки `stats_server`: несколько потоков, каждый использует `Logger` с сокет-backend'ом
и отправляет случайные сообщения на заданный адрес.

```bash
./stats_server 127.0.0.1 9000 10 5   # терминал 1
./socket_demo_client 127.0.0.1 9000 3 20 0   # терминал 2 — 3 клиента, по 20 сообщений, без задержки
```

## Ограничения


- `SocketLogStrategy` не реализует переподключение при разрыве соединения — если сервер
  статистики недоступен или разрывает соединение, дальнейшие попытки записи завершаются
  ошибкой (выводится в `stderr`), приложение продолжает работать.
- Формат ввода консольного приложения (Часть 2) не поддерживает сообщения, которые должны
  начинаться буквально с зарезервированного слова уровня (`DEBUG`/`INFO`/`WARNING`/`ERROR`) —
  такое сообщение будет ошибочно интерпретировано как явное указание уровня.
